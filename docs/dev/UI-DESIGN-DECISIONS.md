## 架构决策

### 决策1: 采用状态驱动的终端界面

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
原始设计中，终端界面是静态的，未连接和已连接状态使用相同的布局，导致用户在未连接时面对大量空的或禁用的控件。

**决策：**
采用状态驱动设计，终端界面根据连接状态（未连接/已连接/错误）动态调整布局和内容。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 静态布局 | 实现简单 | 未连接时空面板多，用户困惑 | ❌ |
| B. 状态驱动 | 清晰的状态转换，更好的用户体验 | 实现复杂度略高 | ✅ |
| C. 多个独立页面 | 逻辑分离 | 页面切换繁琐，上下文丢失 | ❌ |

**理由：**
1. **渐进式复杂度**：新用户看到简化的连接向导，专家用户看到完整的监控面板
2. **减少认知负荷**：未连接时只显示连接相关的控件，已连接时显示数据监控
3. **提高效率**：一次点击完成连接，而不是"配置→应用→连接"的多步流程

**权衡：**
- 增加了状态管理复杂度
- 需要精心设计状态转换动画
- 需要确保状态一致性（如后台断开连接）

**实施细节：**
```tsx
// 伪代码示例
const TerminalWorkspace = () => {
  const { connectionStatus } = useConnection()

  switch (connectionStatus) {
    case 'disconnected':
      return <QuickConnectWizard />
    case 'connected':
      return <DataMonitorWorkspace />
    case 'error':
      return <ConnectionErrorState />
    default:
      return <LoadingState />
  }
}
```

**后续影响：**
- 所有终端界面组件需要支持三种状态
- 状态转换需要动画过渡
- 错误处理需要统一规范

### 决策2: 整合协议配置到终端界面

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
原始设计中，协议配置是独立的"协议管理"界面，用户需要：终端界面 → 协议界面配置 → 返回终端界面，工作流断裂。

**决策：**
将协议配置降级为终端界面的辅助功能，"协议管理"界面改为"协议模板库"，仅用于管理预定义协议。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 独立协议管理界面 | 功能集中 | 跨界面操作，上下文切换 | ❌ |
| B. 整合到终端界面 | 工作流完整，不离上下文 | 终端界面略复杂 | ✅ |
| C. 协议配置作为全局设置 | 一次配置，所有端口可用 | 不支持每个端口不同协议 | ❌ |

**理由：**
1. **上下文完整性**：用户在终端界面配置协议后可以立即测试，无需切换
2. **符合用户心智模型**：协议是绑定到端口和会话的，不是全局配置
3. **提高效率**：减少界面切换次数

**权衡：**
- 终端界面右侧辅助面板功能增多（从3个卡片增加到4个）
- "协议管理"界面需要重新定位为"协议模板库"
- 协议配置数据模型需要调整（从全局改为会话级别）

**实施细节：**
```tsx
// 右侧辅助面板新增"协议控制"卡片
<ProtocolControlCard>
  <CurrentProtocol value={modbusRTU} />
  <QuickConfigure />
  <AddNewProtocol />
</ProtocolControlCard>

// 点击"配置"打开抽屉（不是模态窗口）
<ProtocolConfigDrawer>
  <ModbusRTUConfig />
</ProtocolConfigDrawer>
```

**后续影响：**
- "协议模板库"界面使用频率降低，需要明确其定位
- 协议配置需要支持端口级别（不同端口可配置不同协议）
- 需要支持协议配置的导入/导出

### 决策3: 脚本编辑器采用可调整分栏布局

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
原始设计中，脚本列表、编辑器、输出区固定为20%:50%:30%，但输出区在95%的时间里是空的或只有很少内容，浪费空间。

**决策：**
采用可调整分栏布局，允许用户拖动分隔线自定义比例，输出区默认折叠（仅显示标题栏）。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 固定比例 | 简单，不会破坏布局 | 空间浪费，不灵活 | ❌ |
| B. 可调整分栏 | 灵活，用户自定义 | 实现复杂，可能破坏布局 | ✅ |
| C. 自动调整（AI判断） | 智能化 | 不可预测，用户困惑 | ❌ |

**理由：**
1. **真实使用场景**：用户95%时间在编写脚本，只有5%时间查看输出
2. **用户偏好差异**：有的用户喜欢大编辑器，有的喜欢大输出区
3. **提高效率**：更大的编辑器区域，更好的代码编写体验

**权衡：**
- 需要实现拖动分隔线逻辑
- 需要保存用户偏好（localStorage）
- 移动端适配复杂（分栏在小屏幕不适用）

**实施细节：**
```tsx
<SplitPane
  direction="horizontal"
  defaultSizes={[20, 50, 30]}  // 默认比例
  minSizes={[150, 300, 60]}     // 最小尺寸（输出区可折叠到60px）
  onSave={saveUserPrefs}        // 保存用户偏好
>
  <ScriptList />
  <MonacoEditor />
  <OutputPanel collapsible defaultCollapsed />
</SplitPane>
```

**后续影响：**
- 响应式设计需要特别处理（移动端不支持分栏）
- 需要提供"重置布局"按钮，恢复默认比例
- 需要测试极端比例（如列表10%，编辑器80%，输出区10%）

## 交互设计决策

### 决策4: 循环发送使用独立工具抽屉

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
循环发送是危险操作（可能刷爆设备），但如果将其隐藏在模态窗口中，用户又难以监控状态。

**决策：**
循环发送配置使用独立的工具抽屉（从底部滑出），运行时在顶部状态栏和发送区提供强烈的视觉警告。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 模态窗口 | 聚焦注意力 | 阻断用户，难以监控 | ❌ |
| B. 工具抽屉 | 不阻断，可随时调整 | 占用屏幕空间 | ✅ |
| C. 侧边栏卡片 | 始终可见 | 挤占其他卡片空间 | ❌ |

**理由：**
1. **安全性优先**：强烈的视觉警告（橙色脉冲边框）让用户随时意识到循环发送在运行
2. **可监控性**：不阻断主界面，用户可以观察RX区的响应
3. **全局紧急停止**：⌘.快捷键在任何位置都可停止

**权衡：**
- 工具抽屉占用底部空间（约200px高度）
- 需要设计多状态指示（未启动/运行中/暂停）
- 需要防止误操作（如误勾选循环发送）

**实施细节：**
```tsx
// 发送区
<SendPanel>
  <Input />
  <Actions>
    <Button>发送</Button>
    <Button onClick={openCyclicSendDrawer}>⚙️发送设置</Button>
  </Actions>
</SendPanel>

// 循环发送工具抽屉
<CyclicSendDrawer position="bottom">
  <Toggle enabled={cyclicEnabled} />
  <Interval min={10} max={5000} value={interval} />
  <StopCondition type="manual|count|time" />
  <CurrentStatus count={1234} />
  <Actions>
    <Button>▶开始</Button>
    <Button>⏸暂停</Button>
    <Button>■停止</Button>
  </Actions>
</CyclicSendDrawer>

// 运行时的视觉警告
<TopBar className={cyclicRunning ? 'pulse-orange' : ''}>
  {cyclicRunning && (
    <>
      <Badge variant="warning">🔄循环发送中 {count}次</Badge>
      <Button onClick={emergencyStop}>■紧急停止</Button>
    </>
  )}
</TopBar>

<SendPanel className={cyclicRunning ? 'pulse-orange-border' : ''} />
```

**后续影响：**
- 需要设计紧急停止的全局快捷键（⌘.）
- 需要处理循环发送时的并发问题（如用户手动发送）
- 需要记录循环发送历史（防止意外刷爆设备）

### 决策5: 使用抽屉而非模态窗口进行配置

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
原始设计中，所有配置都使用模态窗口，但模态窗口会阻断用户操作，不适合需要参考上下文的配置（如协议配置、发送设置）。

**决策：**
建立模态窗口和抽屉的使用规范：
- **模态窗口**：多步骤向导、危险操作确认、文件操作
- **抽屉**：快速配置、详情查看、上下文工具

**选项对比：**

| 选项 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 模态窗口 | 聚焦注意力 | 阻断操作，不能参考上下文 | 创建端口、导出数据 |
| 抽屉 | 不阻断，可参考主界面 | 占用屏幕空间 | 协议配置、发送设置 |
| 侧边栏 | 始终可见，不占额外空间 | 挤占主界面空间 | 端口详情、数据统计 |

**理由：**
1. **上下文保持**：配置协议时可以参考RX区的数据
2. **快速调整**：配置发送参数时可以立即测试
3. **减少点击**：不需要"打开配置→应用→关闭"的多步流程

**权衡：**
- 抽屉占用屏幕空间，需要精心设计尺寸
- 需要明确何时使用模态窗口vs抽屉
- 移动端抽屉可能占满整个屏幕

**实施细节：**
```tsx
// 模态窗口：创建虚拟端口对
<Modal open={isOpen} onClose={() => setIsOpen(false)}>
  <CreateVirtualPortForm />
</Modal>

// 抽屉：配置协议
<Drawer open={isOpen} onClose={() => setIsOpen(false)} position="right">
  <ProtocolConfig />
</Drawer>

// 抽屉：发送设置（从底部滑出）
<Drawer open={isOpen} onClose={() => setIsOpen(false)} position="bottom">
  <CyclicSendConfig />
</Drawer>
```

**使用规范：**
| 操作类型 | 使用组件 | 示例 |
|---------|---------|------|
| 多步骤向导 | 模态窗口 | 创建虚拟端口对 |
| 危险操作确认 | 模态窗口 | 删除端口、重置应用 |
| 文件操作 | 模态窗口 | 导入/导出配置 |
| 快速配置 | 抽屉 | 协议配置、发送设置 |
| 详情查看 | 抽屉 | 端口详情、错误详情 |
| 上下文工具 | 抽屉 | 搜索、过滤 |

## 视觉设计决策

### 决策6: 采用Cyber-Industrial设计语言

**日期：** 2026-05-11
**状态：** 已采纳（继承v1.0）

**问题描述：**
串口工具的用户主要是工程师和技术人员，需要专业、高效、信息密度高的界面，但不能过于geeky而牺牲易用性。

**决策：**
采用Cyber-Industrial设计语言，结合工业软件的专业性和赛博朋克风格的视觉吸引力。

**核心特征：**
1. **暗色主题**：减少长时间使用的眼睛疲劳
2. **信号色强调**：绿色(#00ff41)作为主要操作色，橙/红作为警告/错误色
3. **高信息密度**：紧凑的布局，丰富的数据展示
4. **清晰的状态指示**：运行状态用脉冲动画、颜色变化表示
5. **等宽字体**：数据显示使用JetBrains Mono，确保对齐

**理由：**
1. **目标用户**：工程师、技术人员，习惯暗色IDE风格界面
2. **使用场景**：长时间监控，需要减少眼睛疲劳
3. **品牌识别**：区别于传统的串口工具（如PuTTY、SecureCRT）
4. **情感共鸣**：赛博朋克风格吸引技术用户

**权衡：**
- 暗色主题在明亮环境下可读性较差
- 高信息密度可能对新用户造成认知负荷
- 绿色信号色可能对色盲用户不友好（需提供无障碍模式）

**颜色系统：**
```css
/* 信号色 - 主要操作 */
--signal-primary: #00ff41;
--signal-dim: rgba(0, 255, 65, 0.08);
--signal-glow: rgba(0, 255, 65, 0.15);

/* 警告色 */
--warning-primary: #ffb142;
--warning-dim: rgba(255, 177, 66, 0.08);

/* 错误色 */
--error-primary: #ff4757;
--error-dim: rgba(255, 71, 87, 0.08);

/* 背景色 */
--bg-deepest: #0d0d0f;
--bg-deep: #121214;
--bg-base: #18181b;
--bg-elevated: #1c1c1f;
```

**字体系统：**
```css
/* 数据显示 */
--font-mono: 'JetBrains Mono', 'SF Mono', monospace;

/* 界面文字 */
--font-sans: 'Instrument Sans', system-ui, sans-serif;

/* 标题 */
--font-display: 'Instrument Serif', Georgia, serif;
```

### 决策7: 使用数据卡片而非表格展示统计信息

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
原始设计中，端口状态、数据统计等信息使用表格展示，但表格对于少量关键指标的展示效率不高。

**决策：**
使用卡片组件展示关键指标（端口状态、数据统计、协议控制），每个卡片包含标题、数值、趋势图。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 表格 | 适合多行数据 | 对于2-3个指标效率低 | ❌ |
| B. 数据卡片 | 视觉突出，支持图表 | 占用空间大 | ✅ |
| C. 仪表盘 | 专业，类似汽车仪表 | 实现复杂，移动端不适用 | ❌ |

**理由：**
1. **视觉层次**：卡片使用阴影和边框，自然形成视觉分组
2. **信息密度**：卡片可以同时显示数值、趋势图、状态指示
3. **可折叠**：用户可以折叠不关注的卡片
4. **响应式**：卡片在小屏幕自动换行，表格需要横向滚动

**实施细节：**
```tsx
// 右侧辅助面板
<div className="auxiliary-panel">
  <Card collapsible defaultOpen>
    <CardHeader>当前端口</CardHeader>
    <CardContent>
      <PortDetail />
    </CardContent>
  </Card>

  <Card collapsible defaultOpen>
    <CardHeader>数据统计</CardHeader>
    <CardContent>
      <Stat label="RX" value={1234} trend="+12%" />
      <Stat label="TX" value={567} trend="+8%" />
      <SparkLine data={trafficData} />
    </CardContent>
  </Card>

  <Card collapsible defaultOpen>
    <CardHeader>协议控制</CardHeader>
    <CardContent>
      <ProtocolSelector />
    </CardContent>
  </Card>
</div>
```

**后续影响：**
- 需要设计卡片的折叠/展开动画
- 需要确保卡片在小屏幕的布局（从垂直变为水平）
- 需要支持卡片的拖拽排序（用户自定义顺序）

## 性能优化决策

### 决策8: RX数据区使用虚拟滚动

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
串口通信可能产生大量数据（如每秒1000包），如果全部渲染会导致内存占用过高和界面卡顿。

**决策：**
RX数据区使用虚拟滚动，只渲染可见区域的数据项，超出部分动态加载和卸载。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 全部渲染 | 实现简单 | 内存占用高，卡顿 | ❌ |
| B. 虚拟滚动 | 内存占用低，流畅 | 实现复杂，需要预估高度 | ✅ |
| C. 分页加载 | 实现简单 | 需要用户手动翻页 | ❌ |

**理由：**
1. **内存控制**：10000行数据全部渲染可能占用500MB+内存
2. **流畅体验**：虚拟滚动确保60fps，即使有大量数据
3. **自动清理**：滚动时自动卸载不可见项
4. **智能缓冲**：上下各预渲染10-20项，减少空白闪烁

**实施细节：**
```tsx
import { VariableSizeList } from 'react-window'

const RxDataViewer = ({ packets }) => {
  const listRef = useRef<VariableSizeList>(null)

  // 估算每行高度（动态计算）
  const getItemSize = (index: number) => {
    const packet = packets[index]
    return packet.data.length > 32 ? 60 : 40  // 长数据行高一些
  }

  return (
    <VariableSizeList
      ref={listRef}
      height={600}
      itemCount={packets.length}
      itemSize={getItemSize}
      width="100%"
    >
      {({ index, style }) => (
        <div style={style}>
          <DataPacketRow packet={packets[index]} />
        </div>
      )}
    </VariableSizeList>
  )
}
```

**权衡：**
- 需要预估每行高度（不准确会导致滚动跳跃）
- 搜索功能需要特殊处理（虚拟滚动后无法Ctrl+F）
- 需要实现"跳转到指定位置"的逻辑

**缓冲策略：**
- 最大保存10000行数据（FIFO清理）
- 虚拟滚动上下各预渲染20项
- 滚动到底部时自动清理前1000行

### 决策9: 脚本输出限制最大行数

**日期：** 2026-05-11
**状态：** 已采纳

**问题描述：**
脚本可能产生大量输出（如循环打印），如果不限制会导致内存溢出和界面卡顿。

**决策：**
脚本输出区限制最大1000行，超过后FIFO自动清理，并显示提示信息。

**选项对比：**

| 选项 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| A. 不限制 | 完整日志 | 内存溢出，卡顿 | ❌ |
| B. FIFO清理1000行 | 内存可控 | 可能丢失重要日志 | ✅ |
| C. 分页显示 | 保留所有日志 | 需要手动翻页 | ❌ |

**理由：**
1. **内存控制**：1000行约占用100-200KB，可接受
2. **自动管理**：用户不需要手动清理
3. **提示友好**：清理前显示"已清理旧日志，保留最新1000行"

**实施细节：**
```tsx
const ScriptOutput = () => {
  const [logs, setLogs] = useState<string[]>([])
  const MAX_LOGS = 1000

  const addLog = (newLog: string) => {
    setLogs(prev => {
      const updated = [...prev, newLog]
      if (updated.length > MAX_LOGS) {
        const removed = updated.length - MAX_LOGS
        console.log(`已清理${removed}条旧日志`)
        return updated.slice(-MAX_LOGS)
      }
      return updated
    })
  }

  return (
    <OutputPanel>
      {logs.map((log, i) => (
        <LogLine key={i}>{log}</LogLine>
      ))}
    </OutputPanel>
  )
}
```

**后续影响：**
- 需要提供"导出日志"功能（完整日志）
- 需要支持日志级别过滤（Error/Warning/Info）
- 需要支持日志搜索（FIFO清理后可能找不到）

## 可访问性决策

### 决策10: 全局快捷键使用⌘而非Ctrl

**日期：** 2026-05-11
**状态：** 已采纳（macOS）

**问题描述：**
应用可能跨平台（macOS/Windows/Linux），需要决定快捷键修饰键。

**决策：**
macOS使用⌘（Command），Windows/Linux使用Ctrl，在代码中自动适配。

**理由：**
1. **平台惯例**：macOS用户习惯⌘，Windows用户习惯Ctrl
2. **避免冲突**：⌘C/Copy在macOS不会与应用功能冲突
3. **一致体验**：遵循各平台的HCI规范

**实施细节：**
```tsx
import { isMac } from '@/utils/platform'

const hotkeys = {
  quickOpen: isMac ? '⌘K' : 'Ctrl+K',
  save: isMac ? '⌘S' : 'Ctrl+S',
  run: isMac ? '⌘Enter' : 'Ctrl+Enter',
  emergencyStop: isMac ? '⌘.' : 'Ctrl+.',
}

// 显示快捷键时
<HotkeyIndicator>
  {isMac ? '⌘K' : '⌘K'}  // macOS显示⌘，Windows显示Ctrl（UI层面）
</HotkeyIndicator>
```

**快捷键映射：**
| 功能 | macOS | Windows/Linux |
|------|-------|--------------|
| 命令面板 | ⌘K | Ctrl+K |
| 快速打开 | ⌘P | Ctrl+P |
| 保存 | ⌘S | Ctrl+S |
| 运行脚本 | ⌘Enter | Ctrl+Enter |
| 紧急停止 | ⌘. | Ctrl+. |
| 清空接收区 | ⌘Shift+X | Ctrl+Shift+X |
